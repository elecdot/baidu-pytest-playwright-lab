\begin{abstract}
本报告记录了我从零搭建百度地图 Web 自动化测试工程的完整实验过程。实验使用
Python Playwright、pytest、pytest-playwright 与 uv 作为主要工具，围绕百度地图
首页、搜索、路线规划、定位权限、移动端视口、HTML 报告、截图和 Trace 诊断能力
展开。与只展示最终代码不同，本报告重点记录实验中的思考路径：为什么选择百度
地图作为被测对象，为什么先做最小 smoke 闭环，为什么在遇到百度安全验证后调整
搜索测试目标，如何通过人工 codegen 观察定位稳定 locator，如何把路线测试从
不稳定交互逐步收敛为 baseline，以及如何用失败演示和重复执行记录证明测试工程
具有可诊断性和一定稳定性。

实验最终形成了 8 个稳定自动化用例，覆盖首页、搜索、路线、定位和移动端基础场景。
稳定套件连续三轮重复执行全部通过，记录分别为 8 passed in 39.94s、8 passed in
41.30s、8 passed in 37.68s。同时，实验保留了百度安全验证、网络波动、headless
与 headed 表现差异等真实问题的证据和分析。本实验不尝试绕过公开网站的安全机制，
也不对地图实时数据进行精确断言，而是把“在真实约束下设计可执行、可维护、可解释
的 UI 自动化测试”作为主要学习目标。
\end{abstract}

\section{概述}

\subsection{选题背景}

本次课程实验要求围绕自动化测试工具完成一个相对完整的工程实践。我选择百度地图
Web 端作为被测对象，主要原因是它不是一个静态页面，而是一个包含地图瓦片、搜索
联想、路线规划、定位权限、动态弹窗、公开站点安全验证等复杂行为的真实 Web 应用。
这类系统比普通登录页、表单页更能暴露端到端 UI 自动化测试中的实际问题：元素
加载顺序不固定、DOM 结构会变化、网络请求可能受限、不同浏览器上下文表现不同，
并且某些流程可能受到反自动化机制影响。

一开始我的目标并不是“把百度地图所有功能都测完”。如果直接把搜索、路线、定位、
移动端、异常输入等所有功能一起实现，失败时很难判断问题来自测试框架、浏览器环境、
locator 不稳定，还是百度地图页面本身的动态行为。因此，本实验采用从小到大的推进
方式：先搭一个最小可运行 smoke 闭环，再逐步加入搜索、路线、定位、移动端和失败
证据。这样每一步都可以运行、截图和回溯，便于把工程问题和被测站点问题分开分析。

\subsection{测试对象}

被测对象为百度地图 Web 端：

\begin{center}
\url{https://map.baidu.com}
\end{center}

百度地图 Web 端的主要特点包括：打开页面时需要加载地图容器和大量脚本资源；搜索
框既支持输入关键词，也支持联想和搜索提交；路线规划面板中存在起点、终点、交通方式
切换和路线详情；定位功能依赖浏览器 geolocation 权限；移动端视口下页面布局会发生
变化。此外，公开站点还存在百度安全验证，提交搜索或访问特定结果页时可能出现图像
旋转、滑块、扫码等验证方式。

\subsection{工具选择思路}

本实验使用 Python Playwright 作为浏览器自动化工具，pytest 作为测试组织框架，
pytest-playwright 作为二者之间的集成层，uv 负责 Python 环境和依赖管理，pytest-html
负责生成 HTML 报告，ruff 和 pre-commit 负责基础代码质量检查。

\decision{选择 Playwright 的原因是它能驱动真实浏览器，支持自动等待、Locator、
截图、Trace、Browser Context、移动端视口和 geolocation 权限模拟；这些能力正好
对应地图类 Web 应用的测试需要。选择 pytest 的原因是它适合把不同阶段的实验用例
拆成独立测试文件，并通过 mark 清晰区分 smoke、search、route、geo 和 mobile。}

\subsection{实验总体目标}

本实验最终希望达到的不是“绕过百度地图限制”，而是完成一套课程层面可展示、可运行、
可解释的自动化测试工程。具体目标如下：

\begin{itemize}
  \item 完成 Python Playwright + pytest + uv 的工程初始化。
  \item 建立 Page Object Model，让测试代码不直接堆砌页面细节。
  \item 形成集中测试数据，便于后续扩展搜索、路线和定位场景。
  \item 完成首页、搜索、路线、定位和移动端的稳定用例。
  \item 为每个关键流程生成截图证据。
  \item 在遇到安全验证、网络波动和 headless/headed 差异时，记录真实现象并调整
  测试策略。
  \item 通过 HTML 报告、失败 Trace 和重复执行结果展示测试工程的可诊断性。
\end{itemize}

\subsection{实验边界}

由于百度地图是公开线上服务，本实验必须遵守合理边界：

\begin{itemize}
  \item 不进行高频请求、批量采集、压力测试或安全扫描。
  \item 不绕过滑块、图像旋转、扫码等百度安全验证。
  \item 不把“出现安全验证”简单视为测试失败，而是作为公开站点约束记录。
  \item 不验证地图数据的绝对准确性。
  \item 不断言实时路线分钟数、公里数、红绿灯数量、价格、路况或路线几何。
  \item 不测试百度地图移动 App，只测试 Web 页面在移动视口下的基础可打开性。
\end{itemize}

这个边界对后续设计非常重要。它决定了测试应该偏向“页面是否能进入目标状态”
和“自动化证据是否完整”，而不是追求对真实地图数据和安全验证流程的完全控制。

\section{测试工具环境建立}

\subsection{项目初始化过程}

项目初始化阶段首先需要解决两个问题：一是让测试代码能够稳定运行；二是让后续实验
过程可以被复现。为此，工程采用了标准 Python 项目结构，并把依赖、测试命令、报告
路径和 agent 运行方式都写入项目配置。这样做的好处是，实验报告中展示的运行命令
不是临时命令，而是和项目实际文件保持一致。

项目初始阶段建立了以下目录：

\begin{itemize}
  \item \path{data/}：集中存放搜索关键词、路线用例和定位坐标。
  \item \path{pages/}：存放 Page Object，封装百度地图页面操作。
  \item \path{tests/}：存放 pytest 用例。
  \item \path{docs/}：存放设计书、locator 观察记录和报告草稿。
  \item \path{reports/}：存放 HTML 报告和 LaTeX 报告工程。
  \item \path{test-results/}：存放截图、Trace 和失败演示证据。
  \item \path{scripts/}：存放 agent-safe 命令包装和失败证据生成脚本。
\end{itemize}

\problem{在 agent 沙箱环境中直接使用 uv 时，默认 cache 路径可能不可写。这个问题
如果不处理，会导致依赖安装、ruff 检查或 pytest 执行不稳定。}

\analysis{我没有把这个问题归因于测试代码，而是先检查项目运行方式。由于仓库内
允许写入，最直接的解决办法是把 uv cache 放到工作区内。这样既不需要修改系统环境，
也不会污染用户主目录。}

\fix{项目增加了 \path{scripts/agent-env.sh}，通过
\texttt{UV\_CACHE\_DIR=.cache/uv} 运行 uv，并在 \path{justfile} 中封装为
\texttt{just agent-ruff}、\texttt{just agent-test} 等命令。}

\subsection{pytest 与 Playwright 配置}

pytest 默认配置包括：

\begin{itemize}
  \item 测试目录：\path{tests}
  \item 默认浏览器：Chromium
  \item 失败时保留 Trace、截图和视频
  \item Playwright 自动产物目录：\path{test-results/playwright}
  \item HTML 报告路径：\path{reports/report.html}
\end{itemize}

这里的配置思路是：正常通过时留下 HTML 报告和主动截图；失败时额外保留 Trace、
截图和视频。这样做能避免“测试失败但没有证据”的问题。对于公开网站自动化测试，
失败原因可能来自网络、安全验证、DOM 变化或测试代码本身，因此失败证据比单纯的
assert 信息更重要。

\subsection{环境验证顺序}

环境建立完成后，我没有立刻编写复杂业务用例，而是采用以下验证顺序：

\begin{enumerate}
  \item 先运行 ruff，确认 Python 代码基本语法和风格没有明显问题。
  \item 再运行最小首页 smoke，确认 Playwright 能打开浏览器并访问百度地图。
  \item 然后逐步加入搜索、路线、定位和移动端测试。
  \item 每增加一类测试，都同步检查截图和 HTML 报告是否生成。
\end{enumerate}

这种顺序的意义在于降低定位成本。如果在所有功能都写完后才第一次运行测试，任何
失败都可能牵涉环境、依赖、页面结构、网络和测试逻辑；而逐步验证可以确保每个阶段
都有一个可回退的稳定点。

\section{测试工具的功能和使用流程}

\subsection{从直接脚本到 Page Object}

最开始实现 smoke 测试时，最简单的写法是在测试函数中直接调用
\texttt{page.goto()}、\texttt{page.locator()} 和 \texttt{page.screenshot()}。
这种写法适合快速验证工具是否可用，但如果后续继续添加搜索、路线、定位和移动端，
测试文件会很快堆满页面细节。为了避免测试代码变成“浏览器操作流水账”，项目引入
了 Page Object Model，把百度地图页面相关操作集中封装到 \texttt{BaiduMapPage}。

首页 smoke 最终在测试中只保留业务含义：

\begin{lstlisting}[language=Python]
baidu_map.goto()
baidu_map.expect_page_loaded()
baidu_map.screenshot("homepage")
\end{lstlisting}

这样做的好处是，当 locator 需要调整时，只修改 \path{pages/baidu_map_page.py}；
测试用例本身仍然表达“打开首页、确认加载、截图”这一意图。对课程实验来说，这也
能更清楚地展示“测试设计”和“页面实现细节”之间的分层。

\subsection{Locator 探索方法}

Playwright 的 Locator 是本实验最核心的能力之一。稳定 locator 的寻找不是一次
完成的，而是经过人工观察、codegen 记录和失败修正逐步确定的。总体原则如下：

\begin{itemize}
  \item 优先使用 role、name、placeholder 等语义 locator。
  \item 当页面缺少语义信息时，再使用 class 或结构 locator。
  \item 不依赖过深的 DOM 层级，避免页面微调导致测试失效。
  \item 对公开站点动态区域，不强行要求某个结果文本必须出现。
\end{itemize}

例如搜索输入框使用：

\begin{lstlisting}[language=Python]
page.get_by_role("textbox", name="搜地点、查公交、找路线")
\end{lstlisting}

路线入口和路线输入框则根据人工 codegen 观察确定：

\begin{lstlisting}[language=Python]
page.locator(".searchbox-content-button")
page.get_by_role("textbox", name="输入起点或在图区上选点")
page.get_by_role("textbox", name="输入终点")
\end{lstlisting}

\problem{搜索和路线页面并不是稳定静态 DOM。搜索提交后可能进入百度安全验证，
路线输入时可能出现候选项，也可能因为 headless 环境表现不同而不出现候选项。}

\analysis{如果测试直接依赖某个候选项或最终结果文本，容易把公开站点的正常动态
行为误判为测试失败。因此 locator 设计要尽量抓住流程入口、输入框、面板和路线
详情这类相对稳定的页面结构。}

\fix{搜索阶段改为“状态分类断言”，路线阶段改为“baseline 验证”：确认路线面板、
方案区域和路线详情区域出现，而不是断言实时路线数值。}

\subsection{自动等待与弹性断言}

Playwright 的 \texttt{expect(locator).to\_be\_visible()} 会自动等待元素达到目标
状态。对公开地图应用而言，断言不能过于精确。本实验采用如下策略：

\begin{itemize}
  \item 首页只断言 body 可见。
  \item 搜索流程不强制断言最终结果页必定出现，而是分类记录结果、安全验证、
  URL 跳转、输入被清空等可观察状态。
  \item 路线规划只断言路线方案区域和路线细节区域出现，不断言实时数值。
  \item geolocation 与 mobile 只验证页面稳定可打开。
\end{itemize}

这里的“弹性”不是降低测试要求，而是让断言和实验目标匹配。比如百度地图路线规划
中，实时路况、路线距离和预计时间都可能随着时间变化。如果断言“必须出现某个固定
公里数”，测试会因为数据变化而失败，无法反映工具能力。相反，断言路线详情面板、
步骤列表和方案区域出现，更能说明 Playwright 成功驱动了路线规划流程。

\subsection{截图、Trace 与 HTML 报告}

项目通过 \texttt{BaiduMapPage.screenshot()} 将关键证据写入
\path{test-results/}。pytest-playwright 负责在失败时保留 Trace、截图和视频；
pytest-html 负责生成 \path{reports/report.html}。

在课程实验中，截图和 Trace 的作用不只是“好看”，而是用于证明每一步确实运行过。
例如首页、搜索输入、搜索安全验证、路线方案、路线详情、定位和移动端都生成了截图。
当测试通过时，截图可以作为结果证据；当测试失败时，Trace 可以回放浏览器操作、
网络请求、DOM 快照和失败时页面状态。

受控失败证据由以下命令生成：

\begin{lstlisting}[language=bash]
just agent-failure-demo
\end{lstlisting}

Trace 查看方式如下：

\begin{lstlisting}[language=bash]
uv run playwright show-trace \
  test-results/failure-demo/failure-demo-trace.zip
\end{lstlisting}

\subsection{Browser Context、定位权限与移动端}

Playwright Browser Context 允许每个测试使用独立的权限、定位、语言、时区和视口。
本实验使用 context 模拟北京定位：

\begin{lstlisting}[language=Python]
browser.new_context(
    geolocation=GEO_LOCATIONS["beijing"],
    permissions=["geolocation"],
    locale="zh-CN",
    timezone_id="Asia/Shanghai",
)
\end{lstlisting}

移动端 smoke 使用 393 x 852 viewport，并开启 \texttt{is\_mobile} 和
\texttt{has\_touch}，用于验证百度地图 Web 页面在移动视口下可以稳定打开。

\subsection{pytest 作为辅助测试工具}

pytest 负责测试发现、用例组织、mark 管理和报告输出。项目按照功能拆分为：

\begin{itemize}
  \item \path{tests/test_home.py}
  \item \path{tests/test_search.py}
  \item \path{tests/test_route.py}
  \item \path{tests/test_geolocation.py}
  \item \path{tests/test_mobile.py}
\end{itemize}

这种拆分方式和实验路线保持一致：\path{test_home.py} 是最小闭环，
\path{test_search.py} 记录反自动化约束下的搜索流程，\path{test_route.py}
记录路线规划 baseline，\path{test_geolocation.py} 和 \path{test_mobile.py}
补充 Browser Context 能力。pytest mark 让我可以按功能运行某一类用例，也可以在
最终阶段运行完整稳定套件。

\section{被测软件介绍}

百度地图 Web 端提供搜索、路线、定位、地图浏览和图层切换等能力。它的页面不是
一次性渲染完成的静态页面，而是随着用户输入、地图视口、网络响应和浏览器权限持续
变化。正因为如此，它适合作为 Playwright 课程实验对象：一方面可以展示真实浏览器
自动化工具的能力，另一方面也会暴露自动化测试在公开复杂站点上的边界。

本实验重点关注以下模块：

\toprule
模块 & 特征 \\
\midrule
\endfirsthead
\toprule
模块 & 特征 \\

\subsection{人工探索记录}

在正式写自动化测试之前，我先通过手工操作和 Playwright codegen 观察页面。这个
步骤非常关键，因为百度地图的输入框、路线入口和结果区域并不一定有简单固定的 id。
如果直接猜 locator，很容易写出只能在某一次运行中生效的脚本。

人工观察主要记录在 \path{docs/locator_discovery_notes.md} 中，过程包括：

\begin{itemize}
  \item 观察首页搜索框的可访问性名称，确认可以使用 role locator。
  \item 观察搜索输入后页面是否出现联想列表。
  \item 观察搜索提交后是否进入地图跳转、安全验证或结果页。
  \item 观察路线入口按钮、路线面板、起点输入框、终点输入框和驾车 tab。
  \item 观察路线规划后页面中哪些区域相对稳定，哪些文本属于实时数据。
\end{itemize}

\problem{人工 codegen 搜索验证中，多次出现“百度安全验证”和“请完成下方验证后
继续操作”。验证方式包括图像旋转、滑块和扫码验证。}

\analysis{这说明搜索流程并不完全由测试代码控制。对于公开站点，安全验证属于
站点保护机制。如果测试目标写成“自动提交搜索并一定进入结果页”，就会和被测站点的
正常安全策略冲突。}

\fix{搜索阶段被调整为 Search Flow Under Anti-Automation Constraint，即在反自动化
约束下验证搜索输入、搜索联想或输入状态、搜索结果状态分类，而不是绕过验证。}

\subsection{Headed 与 Headless 差异}

实验中还观察到 headed 和 headless 模式表现不完全一致。课堂演示时使用 headed
浏览器更容易看到真实页面变化，而 CI 或 agent 执行时一般使用 headless。百度地图
搜索联想、路线候选项和安全验证触发时机都可能受运行模式影响。因此，最终稳定测试
没有依赖“必须出现某个候选项”这样的断言，而是根据可观察状态进行分类或验证更稳定
的页面区域。

这个发现也影响了路线测试的设计。路线规划初期我曾希望精确点击候选项并验证具体
路线，但实际运行发现候选项不一定稳定出现。最终改为直接在路线输入框中输入起点和
终点并按 Enter，再等待路线方案区域和详情区域出现。这种方式更符合公开站点 UI
自动化测试的稳定性要求。

\subsection{核心用例设计说明}

\textbf{首页 smoke。}
首页测试是整个项目的第一条用例。它只做三件事：访问百度地图、断言 body 可见、
保存截图。这个用例看起来很简单，但它验证了浏览器安装、Playwright fixture、
网络访问、截图目录和 pytest 执行链路。如果首页 smoke 不通过，后续搜索和路线测试
都没有意义。

\textbf{搜索流程。}
搜索原计划是“输入关键词、提交搜索、断言结果”。但人工 codegen 和实际运行都发现，
百度安全验证可能在搜索提交后出现。因此搜索测试被拆成三个较稳妥的目标：验证搜索
框能够接受输入；验证输入后页面进入某种可解释状态；验证搜索提交或访问已知结果页
后能够分类记录结果。这样既覆盖了搜索流程，又不把安全验证误判为自动化失败。

\textbf{路线规划。}
路线测试先验证路线入口，再验证驾车路线 baseline。baseline 使用“北京南站到天安门
广场”，原因是这两个地点在百度地图中识别度高，路线结果较容易稳定出现。断言重点
放在路线面板、方案区域和详情步骤，而不是具体公里数和分钟数。

\textbf{定位与移动端。}
定位测试使用 Playwright Browser Context 提供的模拟 geolocation 权限，目标不是
验证百度地图最终定位文本是否绝对准确，而是验证页面能够在授予定位权限的上下文中
稳定加载。移动端测试同理，主要验证 Web 页面在移动 viewport 下不会直接崩溃，并
保存截图作为证据。

\subsection{反自动化约束下的分类断言}

搜索流程不把“出现百度安全验证”视为自动化失败，因为这是公开站点的正常保护
机制。搜索测试的分类结果包括：

\begin{itemize}
  \item \texttt{suggestion}：出现搜索联想。
  \item \texttt{security\_challenge}：出现百度安全验证。
  \item \texttt{map\_navigation}：headless 下提交后发生地图 URL 跳转。
  \item \texttt{input\_cleared}：输入被页面清空。
  \item \texttt{input\_retained\_without\_suggestion}：输入保留但无联想。
  \item \texttt{partial\_input\_retained}：只保留部分输入。
\end{itemize}

这种分类断言是本实验的重要调整。它保留了对搜索流程的观察价值，同时尊重公开站点
安全边界。换句话说，测试通过并不代表“搜索一定成功进入结果页”，而是代表自动化
脚本能够执行搜索相关操作，并把页面最终状态归入已知、可解释、可截图的类别。

\subsection{核心代码}

页面操作集中在 \texttt{BaiduMapPage}。例如首页访问和截图方法承担了所有测试的
公共基础能力：

\begin{lstlisting}[language=Python]
def goto(self) -> None:
    self.goto_url(self.BASE_URL)

def expect_page_loaded(self) -> None:
    expect(self.page.locator("body")).to_be_visible()

def screenshot(self, name: str) -> Path:
    TEST_RESULTS_DIR.mkdir(parents=True, exist_ok=True)
    path = TEST_RESULTS_DIR / f"{name}.png"
    self.page.screenshot(path=path, full_page=True)
    return path
\end{lstlisting}

路线规划 baseline 使用以下 Page Object 方法：

\begin{lstlisting}[language=Python]
def plan_drive_route(self, start: str, end: str) -> None:
    self.open_route_panel()
    self.drive_mode_tab.click()
    self.route_start_input.click()
    self.route_start_input.press_sequentially(start, delay=120)
    self.route_start_input.press("Enter")
    route_end_input = self.route_end_input.first
    route_end_input.click()
    route_end_input.press_sequentially(end, delay=120)
    route_end_input.press("Enter")
\end{lstlisting}

稳定性修复中，页面导航增加了最多 3 次重试，用于缓解公开站点偶发的网络变化错误：

\begin{lstlisting}[language=Python]
def goto_url(self, url: str) -> None:
    last_error = None
    for _ in range(3):
        try:
            self.page.goto(url, wait_until="domcontentloaded")
            return
        except PlaywrightError as error:
            last_error = error
            self.page.wait_for_timeout(1000)
    raise last_error
\end{lstlisting}

\problem{重复执行稳定性测试时，曾出现一次 \texttt{net::ERR\_NETWORK\_CHANGED}。
这类错误通常不是断言问题，而是公开站点访问或本地网络状态瞬时变化造成的导航失败。}

\analysis{如果把这个问题简单归为测试失败，后续重复执行会受偶发网络影响。另一方面，
也不能无限重试，否则会掩盖真实问题。}

\fix{对页面导航增加有限重试，最多 3 次，每次失败后等待 1 秒。这样既提高了公开
站点端到端测试的鲁棒性，又不会让失败被无限隐藏。}

\problem{路线结果测试中曾被地图比例尺里的“公里”误触发，导致断言没有真正落在路线
详情上。}

\analysis{地图页面上同一个词可能出现在多个区域，例如比例尺、路线说明、广告卡片
或其他控件。直接用全页面文本匹配容易产生误判。}

\fix{后续断言改为围绕路线规划结果区域和详情区域进行，不把全页面任意位置出现的
“公里”当作路线成功证据。}

诊断能力。这样设计的原因是：如果只提交全部通过的测试，老师只能看到成功结果，
但看不到工具在失败时能提供哪些调试信息。UI 自动化测试的实际价值不仅在于发现
通过，也在于失败时能解释“失败发生在什么时候、页面是什么状态、浏览器做过哪些
动作”。

\problem{真实失败往往不可控，例如网络断开或页面安全策略变化。如果为了报告临时
等待真实失败，结果可能不可复现。}

\analysis{课程实验需要稳定展示失败诊断能力，因此更合理的方式是构造一个受控失败：
正常打开百度地图，然后等待一个确定不存在的文本。这样失败原因清楚，Trace 也可以
稳定生成。}

\fix{失败演示脚本输出截图、Trace 和 summary 文件，并且不纳入 8 个稳定 pytest
用例。因此它能展示失败证据能力，同时不会影响稳定套件通过率。}

\section{总结}

\subsection{实验收获}

通过本次实验，我对 Web UI 自动化测试的理解从“写脚本点击页面”转向了“设计可维护
的测试工程”。一开始我更关注能否把百度地图某个功能跑通；后面逐步发现，真正影响
实验质量的是测试目标是否清楚、locator 是否可靠、断言是否贴近业务状态、失败是否
有证据、命令是否可复现。

本实验中最重要的经验有三点。第一，复杂公开站点不能只按理想流程设计测试，必须
接受安全验证、网络波动和动态 DOM。第二，自动化测试不是越精确越好，实时地图数据
和路线数值不适合硬编码断言。第三，报告和证据链是测试工程的一部分，HTML 报告、
截图、Trace、重复执行记录能让测试结果更容易被检查和复现。

\subsection{Playwright、pytest 与 uv 的优点}

\begin{itemize}
  \item Playwright 的 Locator、自动等待、Browser Context、截图和 Trace 能力
  适合复杂 Web UI 的端到端验证。
  \item pytest 提供了清晰的用例组织、mark 分类、fixture 复用和 HTML 报告扩展。
  \item uv 提供了快速、可复现的 Python 依赖和命令运行环境。
  \item Page Object Model 降低了测试用例与页面细节的耦合。
\end{itemize}

\subsection{不足与经验}

\begin{itemize}
  \item 公开地图站点存在安全验证，完整搜索流程不能强行自动化。
  \item Headed 和 headless 行为存在差异，例如搜索联想和路线候选项表现不同。
  \item 外部网络波动会影响端到端测试稳定性，需要重试和失败证据。
  \item 地图瓦片、实时路况、路线时长等动态数据不适合精确断言。
\end{itemize}

这些不足并不意味着 Playwright 不适合测试百度地图，而是说明测试工具和测试目标
需要匹配。对公开地图应用来说，更合适的目标是验证关键用户路径、记录状态分类、
保留证据和发现回归；不合适的目标是绕过安全机制或断言实时地图数据的精确值。

\subsection{与 JMeter、Wireshark、Postman 的互补关系}

Playwright 关注用户视角的浏览器端流程，适合验证页面交互和端到端用户路径。
JMeter 更适合接口或服务端压力测试；Wireshark 更适合网络抓包和协议层分析；
Postman 更适合 API 调试、接口集合管理和接口回归测试。对于复杂 Web 应用，四类
工具并非替代关系，而是覆盖不同测试层次：Playwright 验证真实浏览器行为，
Postman 验证接口契约，JMeter 验证负载能力，Wireshark 分析网络通信细节。

\subsection{最终结论}

本实验完成了一套可执行、可诊断、可复现的百度地图 Web 自动化测试工程。最终
稳定套件覆盖首页、搜索、路线、定位和移动端，形成 HTML 报告、截图和 Trace
证据。实验表明，Python Playwright + pytest + uv 适合复杂地图类 Web 应用的
工程化自动化测试，但测试设计必须尊重公开站点安全边界，避免高频请求和过度
精确断言。

从课程实验角度看，本项目的价值不只在于最终 8 个用例通过，而在于完整展示了从
项目初始化、最小闭环、人工 locator 探索、安全验证调整、路线 baseline 收敛、
定位和移动端补充、失败证据演示到稳定性记录的全过程。这个过程说明自动化测试工程
需要不断根据真实页面反馈调整方案，而不是一次性写出最终脚本。
